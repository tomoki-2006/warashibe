# 2. engine単体テストケース表（実装必須の最低ライン）

## exchange.ts / evaluateOffer
- 正解アイテム→accept・gained正しい・hintLevelShown=現状維持
- 不正解1回目→decline・L1文言・rejections=1
- 不正解2回目→L2（ぶん文言）／3回目→L3
- L3以降の不正解→L3を維持（4段目が存在しないこと）
- 提示済みアイテム→duplicate（rejections増えない・ヒント進まない）
- 質問使用回数はofferに影響しない

## recipe.ts / combine
- (wara, abu)→toy／(abu, wara)→toy（順不同）
- 不一致ペア→null／同一アイテム2つ→null／存在しないID→null

## score.ts
- 質問0・ヒント0→100／質問2→70／L1のみ→70／L2→50／L3→20（質問0時）
- 質問2＋L3→下限10で止まる（マイナスにならない）
- ランク境界値: 539/540（大商人/長者）・419/420（行商人/大商人）・249/250（見習い/行商人）※v3.2改訂値・観察ボーナス+10の加算検証を含む

## progress.ts
- applyOffer成立→inventory入替・cleared=true・zukan記録・次ストップ解放
- advanceStop最終→RouteClear状態
- 全操作でイミュータブル（元stateの不変検証）

## validate.ts
- 正常データ→エラー0
- gives連鎖を1箇所切断→到達不能エラー検出
- valueForNpc≦baseValueを仕込む→検出／41字の文→検出／ルビ欠落漢字→検出

# 5. 子どもプレイテスト・プロトコル（Phase 0ゲート判定の正式手順）

## 5.1 リクルートと倫理
- 対象: 小1〜小6の親子・**2巡合計30組以上**（コア小3-4を過半に）。**知人経由は50%上限**、残りは学童・子ども食堂・地域SNS経由（謝礼同額）。n<30の場合は「判定保留」とし追加リクルート（v3.2）
- 親の書面同意（録画は手元と画面のみ・顔なし・データは分析後破棄・謝礼は図書カード等少額）
- 子どもへの声かけ台本統一: 「これ、まだ作りかけのゲームなんだ。難しかったらそれはゲームのせいだから教えてね」（**子どもに失敗の帰属をさせない**）

## 5.2 セッション設計（1組40分）
1. 導入2分（台本どおり）
2. プレイ20分: **一切助けない。** 詰まっても「どうしたらいいと思う？」の返しのみ
3. もう一回遊ぶ？と聞かずに端末を置いたまま3分雑談（自発的再プレイの観測窓）
4. 子どもインタビュー5分: 「どこがいちばん楽しかった？」「ぶんのこと、どう思う？」（好き/こわい/どちらでもない）
5. 親インタビュー5分: 「お子さんに何が起きていたと思いますか」「家で遊ばせたいですか（5段階）」「地方パック¥730なら？ おやこプレミアム月¥380なら？」
6. 対面不能時の代替: ビデオ会議観察＋保護者スマホ2台撮り（手元/画面）プロトコル。観察精度の差は記録に明記（v3.2）
7. 運営者が海外在住の場合（v3.3）: **現地の日本語補習校・日系コミュニティ経由で日本語話者の子どもをリクルートすれば週末に対面テストが組める**（PWAなのでURL共有だけで即実施可）。日本側の巡はマーケ委託または家族の協力＋ビデオ観察で並走

## 5.3 観察シート（1ストップ1行で記録）
| 記録項目 | 例 |
|---|---|
| 初回提示までの秒数 | 34s |
| 質問使用 | 1 |
| 誤提示回数/到達ヒント段 | 1回/L1 |
| 発話・表情 | 成立時に「よっしゃ」 |
| 離脱兆候 | 視線が親へ・姿勢崩れ |

## 5.4 判定（v2.0 12.1のゲートを操作的に定義）
- 完走率はゲートから除外し**参考値**とする（詰みゼロ設計下では指標として無情報・v3.2）
- 自発再プレイ = 観測窓3分の再開は参考値。**判定は「配布後48時間の自然再起動 ≥ 35%」**（T-032テレメトリで計測）
- D1/D2 = テレメトリの自然起動で判定 ≥ 30%（**親アンケートは廃止**——謝礼を受けた知人の自己申告は偽陽性バイアス）
- **盲検化**: 観察記録者にはゲート基準を伏せる。Kaiは判定会議まで個票・生データに触れない（v3.2）
- ぶん好感 = 「好き」が70%以上かつ「こわい」20%未満（未達で雀プランB発動）
- **ヒントL3到達率がストップ単位で20%超 → 該当ストップは難度バグとして即修正対象**

# 6. バグ運用

- 重大度: S1=進行不能・データ消失（即hotfix）／S2=誤動作・演出破綻（週内）／S3=軽微（バックログ）
- 子どもテスト中に発見のS1は当日修正→翌日再テスト（preview URL即時反映の運用が効く）
- 既知バグ一覧は `docs/KNOWN_ISSUES.md` に常時最新化（テスト協力親への誠実さ＝口コミの種）
