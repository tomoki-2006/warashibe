# 第1部 戦略 — なぜこれが天下を取れるのか

## 1.1 市場の空白

- 英語学習 → Duolingoが王者
- 算数・計算 → トドさんすう等が占有
- **お金・経済×子ども → 王者不在。** 2022年から高校家庭科で金融教育が必修化されたが、小中学生向けの「面白いお金ゲーム」は存在しない。親の関心（金融教育への不安）と供給の間に巨大なギャップがある。
- 桃鉄教育版が学校導入で成功したことで、「ゲームで経済を学ぶ」への学校側の心理的ハードルは既に下がっている。先行者が耕した畑に、より深い作物を植える。

## 1.2 勝ち筋（4段ロケット）

1. **Phase 0-1: 面白さの証明** — 純粋なゲームとして子どもが自発的に繰り返し遊ぶことを実証。教育の看板は一切出さない。
2. **Phase 2: 親への信頼獲得** — 広告なし・ガチャ課金なし・親ダッシュボードで「何を学んだか」を可視化。親の口コミ（ママ友ネットワーク）が最強の獲得チャネル。
3. **Phase 3: バイラルループ** — 旅の結果カード（藁→ミカン→…→屋敷の成長チェーン）をSNS共有。Wordle型の「結果だけ見せる」共有。
4. **Phase 4: 学校・教育機関チャネル** — 桃鉄教育版と同じ順番。面白さが証明された後に教育版ライセンス。金融教育必修化の追い風をB2B2Cで刈り取る。

## 1.3 守りの設計（親の信頼＝参入障壁）

競合が真似できない/しにくい点を最初から埋め込む:

- **課金ガチャ完全排除**（コレクション要素はプレイでのみ獲得）
- **広告ゼロ**
- **チャットなし設計**（子ども安全。共有ワールドでも定型文交渉のみ）
- **確率・仕組みの完全開示**
- **プレイ時間の親側コントロール**（1日の旅回数上限を親が設定可能）

この「親が安心して渡せる」設計自体がブランドになり、F2P競合が構造的に真似できない堀になる。

# 第2部 プロダクト設計

## 2.1 ターゲット

| 層 | 年齢 | 設計上の対応 |
|---|---|---|
| コア | 小3〜小6（8-12歳） | メインバランスはここに合わせる |
| 下限拡張 | 年長〜小2 | 全文ふりがな＋音声読み上げ＋「よみあげモード」 |
| 上限拡張 | 中学生〜大人 | 後半時代（株・信用）と実績システムが受け皿 |

**コア年齢の決定: 小3〜小6に確定する。** 理由: (a) 物々交換→通貨の抽象化を理解できる最低ライン、(b) 自分のタブレット/親のスマホ接触時間が最大、(c) 桃鉄教育版の主戦場と同じ＝学校チャネルの導線が既にある。

## 2.2 コアループ（1回の旅 = 15〜20分）

```
出発（藁を受け取る）
  ↓
【観察】NPCのセリフ・見た目・状況から「困りごと」を読む
  ↓
【推理】困りごとを解くモノは何か？ 手持ちで解けるか？
  ↓
【移動】解けないなら、そのモノがある場所へ地図で移動（＝地理）
  ↓
【交換】提案 → 成立/丁寧な拒否（ヒント付き）
  ↓
価値の階段を1段上る → 次のNPCへ
  ↓
ゴール（屋敷）→ 結果カード生成 → 図鑑登録
```

### コアスキルは「交渉」ではなく「観察」

値切りゲームにしない。**相手をよく見て、何に困っているかを読む**のが上手さ。これは経済学の本質（価値は主観的・取引は相手の問題解決）であり、同時に共感力のトレーニングでもある。親に説明できる教育的支柱がここ。

- NPCに「質問する」コマンド（1人につき2回まで）。質問で困りごとのヒントが出る
- 間違ったモノを提案 → NPCは怒らず、丁寧に断りつつヒントを1つ出す
- 3回断られたら「旅人のささやき」（強めのヒント）が自動発動。**詰み状態を作らない**

### 「あわせ技」メカニクス（クラフト・ライト）

原作の「藁＋アブ＝子どものおもちゃ」を、そのままチュートリアルにする。インベントリで2つのアイテムを組み合わせると新しい価値が生まれる。単体では無価値なものが、組み合わせと相手次第で宝になる——これが第1話で体に入る。

## 2.3 Phase 0 の完全スクリプト: 「吉備の旅」

固定ルート1本。岡山・吉備から姫路方面へ。7ストップ、想定15分。
**このセクションはそのまま実装データになる。**

### ストップ0: 吉備の田んぼ（チュートリアル）
- 登場: おばあさん
- イベント: 転んだプレイヤーが藁を1本掴む。「それも何かの縁じゃ。大事にしんさい」
- 獲得: `item_wara`（藁）
- 教える操作: 移動、インベントリ、話しかける

### ストップ1: 吉備津神社の参道
- 登場: 泣いている子ども＋困った母親、ブンブン飛ぶアブ
- 困りごと（明示しない）: 子どもは退屈で泣いている。母は参拝が終わらず困っている
- 観察ヒント: 子どもの視線がアブを追っている（アニメーションで表現）
- 正解手順: アブを捕まえる（タップミニゲーム、失敗なし）→ インベントリで `藁+アブ` を組み合わせ → `item_abumushi_toy`（アブつき藁のおもちゃ）→ 子どもに渡す
- 見返り: 母親から `item_kibidango`（吉備団子）「お礼にこれを。吉備の名物ですよ」
- 学び（裏）: 価値は主観的。大人にはゴミ、子どもには宝物。＋岡山名物きびだんご
- 誤提案時の反応例: 藁だけを差し出す →「ただの藁じゃ、この子は喜ばんよ…」（ヒント: 子どもが見ているものに注目）

### ストップ2: 峠の茶屋
- 登場: 腹ぺこの行商人（倉敷へ急いでいる）
- 困りごと: 昼を食べ損ねて力が出ない。だが荷物が多く茶屋に寄る時間もない
- 正解: 吉備団子を渡す
- 見返り: `item_bizenyaki`（備前焼の小鉢）「備前の窯元で仕入れた品じゃ。焼き物は割れんように持って行きなされ」
- 学び（裏）: 備前焼＝岡山の伝統工芸。行商人という職業の存在
- 地理演出: ここで地図が開き、倉敷の位置と「繊維の町」の予告が入る

### ストップ3: 倉敷の川辺
- 登場: 布問屋の娘
- 困りごと: 大事な茶会で使う器を割ってしまった。今日中に代わりが要る
- 質問で出るヒント: 「お客様は焼き物にうるさい方で…」
- 正解: 備前焼の小鉢
- 見返り: `item_hanpu`（倉敷帆布の反物）「うちで織った帆布です。丈夫さは日本一ですよ」
- 学び（裏): 倉敷＝繊維の町（帆布・児島デニムの源流）

### ストップ4: 瀬戸内の港
- 登場: 漁師（船の帆が破れて漁に出られない）
- 正解: 帆布の反物
- 見返り: `item_tai`（立派な鯛）＋船で対岸ショートカット（移動演出）
- 学び（裏）: 瀬戸内海の漁業。布→帆という用途の変換

### ストップ5: 山陽道の宿場
- 登場: 病気の馬を持て余す侍（急ぎの任務、動かない馬に困り果てている）
- 困りごとの二段構造: 侍は「早く行きたい」、馬は「弱っている」
- 正解: 鯛を渡す（侍の旅の食料に）→ 侍は徒歩で出発、馬を託される
- 追加シーケンス: 馬の世話ミニイベント（水を飲ませる場所を地図から選ぶ）→ 馬が回復 `item_uma`
- 学び（裏）: 手間をかけると価値が上がる（労働が価値を生む）

### ストップ6: 姫路への街道 → ゴール
- 登場: 大店の主人（長旅に馬が必要、来春まで留守にする屋敷がある）
- 正解: 馬を貸す/譲る
- 見返り: `item_yashiki`（屋敷）— ゴール！
- エンディング: 藁→おもちゃ→団子→備前焼→帆布→鯛→馬→屋敷 の成長チェーンが一本のアニメーションで流れる
- 結果カード生成（→ 2.6）

### 感情設計の基準

ストップ1（泣く子ども）は感情のピークとして特に丁寧に作る。「困っている人を助けたら、思いがけないお礼が返ってきた」という体験の原型がここ。**教育アプリの天下は、機能ではなく「最初の15分の感情」で決まる。**

## 2.4 ゲームシステム仕様

### 2.4.1 欲望リーディング（Desire Engine）

各NPCは以下のデータ構造を持つ:

```json
{
  "npc_id": "npc_crying_child",
  "name": "泣いている子ども",
  "location": "loc_kibitsu_shrine",
  "problem": "退屈で泣いている",
  "desire_hidden": true,
  "accepts": [
    { "item": "item_abumushi_toy", "reaction": "accept_delight",
      "gives": "item_kibidango", "giver": "npc_mother" }
  ],
  "rejects_default": {
    "reaction": "polite_decline",
    "hint_ladder": [
      "この子が何を見ているか、よく見てみよう",
      "ブンブン飛んでいるものと、手元の藁…",
      "アブを捕まえて、藁と組み合わせてみたら？"
    ]
  },
  "questions": [
    { "q": "どうして泣いているの？", "a": "「たいくつなの！じんじゃ、つまんない！」" },
    { "q": "何が好き？", "a": "「うごくもの！ぴょんぴょんするの！」" }
  ]
}
```

- `hint_ladder`: 断られるたびに1段強いヒント。3段目で実質答え。**詰みゼロ保証**
- 質問は1NPC 2回まで（無料・ノーペナルティ。観察を促す設計）

### 2.4.2 価値の可視化（アハ演出）

交換成立の瞬間に「価値メーター」を見せる:

```
きびだんご
  あなたにとって:    ★★☆☆☆
  行商人にとって:    ★★★★★ ←「はらぺこだから！」
```

**同じモノの価値が相手によって違う**ことを、毎回の交換で視覚的に刷り込む。これがこのゲーム最大の教育装置。説明文はゼロ。星の差だけ見せる。

### 2.4.3 目利き度（スコア）【廃止・v3.2】

> **本節は旧案。スコアの数式・ランク閾値は 01_GDD §4 が唯一の正**（減点内部処理・採点6ストップ・最大600）。以下は履歴として残す。

- 初回提案で成立: +100
- 質問を使って成立: +70
- ヒント1〜2段で成立: +40
- ヒント3段（答え）で成立: +10
- 旅の最後に合計 →「目利き度」ランク（見習い/行商人/大商人/長者）
- **失点表示はしない。加点のみ。** 低ランクでもクリアは同じ。再挑戦の動機として機能させる

### 2.4.4 図鑑（Phase 1〜）

- 特産品図鑑: 交換で手にしたモノが登録される。産地・豆知識付き
- 人物図鑑: 出会ったNPC。職業の説明付き（行商人とは、問屋とは）
- 地図の解放: 旅をクリアすると隣接地方が解放される

### 2.4.5 デイリー行商（Phase 1〜、継続の仕掛け）

- 1日1回、5分の短い交換クエスト（1〜2交換のみ）
- 連続日数で「行商人の暖簾」が育つ（視覚的成長・失っても翌日から再開可能。Duolingoのstreakの優しい版）
- **煽り通知はしない。** 通知文言は「市場が開きました」のみ、時間帯は親が設定

### 2.4.6 結果カード（バイラル装置）

旅クリアで自動生成される1枚画像:

```
┌─────────────────────────┐
│  🌾 → 🧸 → 🍡 → 🏺 → 🧵 → 🐟 → 🐴 → 🏠  │
│                                          │
│   かいの「吉備の旅」                      │
│   目利き度: 大商人 ★★★☆                │
│   6回の交換で わら → 屋敷                │
│                                          │
│   #わらしべワールド                       │
└─────────────────────────┘
```

- 個人情報なし・成長チェーンのみ。親がSNSに載せたくなるデザイン
- 「うちの子、藁から屋敷まで行った」は親の自慢テンプレとして機能する

### 2.4.7 親ダッシュボード（Phase 2、課金転換の要）

- 今週の学び: 「価値の主観性を3回体験」「倉敷＝繊維の町を学習」等を平易な日本語で
- プレイ時間・旅回数の上限設定
- 子どもの目利き度の推移グラフ
- **ここが親を払う気にさせる画面。** 子どもは面白さで遊び、親は成長の可視化で払う

## 2.5 画面仕様（Phase 0）

| # | 画面 | 要素 |
|---|---|---|
| S1 | タイトル | ロゴ、はじめる/つづきから、音声ON/OFF |
| S2 | マップ | 横スクロールの街道絵巻（2D）。現在地、行き先タップ移動、ストップのアイコン |
| S3 | 会話 | NPC立ち絵、セリフ（全文ふりがな）、選択肢: [はなす][しつもん][こうかん] |
| S4 | インベントリ | 所持品グリッド、[くみあわせる]ボタン、アイテム詳細（産地・豆知識） |
| S5 | 交換提案 | 手持ちから選んで差し出す → 成立演出 or 丁寧な拒否＋ヒント |
| S6 | 価値メーター | 交換成立後のアハ演出（2.4.2） |
| S7 | リザルト | 成長チェーンアニメ → 結果カード → [もういちど][シェア] |

- アートは**絵巻物風の温かい2D**（水彩調）。3D禁止。既存IPに似せない
- 全テキスト読み上げ対応（Web Speech API）
- 操作はタップのみ。時間制限なし。「考える時間」を奪わない

## 2.6 データモデル（実装用）

```typescript
// アイテム
interface Item {
  id: string;              // "item_kibidango"
  name: string;            // "きびだんご"
  emoji: string;           // 図鑑・カード用
  origin: string;          // "岡山県"
  trivia: string;          // 図鑑の豆知識（ふりがな付き）
  baseValue: number;       // 1-10（演出用。ゲームロジックは accepts で決まる）
}

// 組み合わせレシピ
interface Recipe {
  inputs: [string, string];    // ["item_wara", "item_abu"]
  output: string;              // "item_abumushi_toy"
  discoveryHint?: string;
}

// 場所
interface Location {
  id: string;
  name: string;
  region: string;           // "岡山" | "倉敷" | ...
  mapX: number;             // 絵巻上の座標
  npcs: string[];
  unlockedBy?: string;      // 先行ストップのクリア
}

// 旅（ルート）
interface Route {
  id: string;               // "route_kibi_01"
  title: string;            // "吉備の旅"
  startItem: string;
  goalItem: string;
  stops: string[];          // location idの順序
  estimatedMinutes: number;
}

// セーブデータ（Phase 0はlocalStorage）
interface SaveData {
  playerName: string;
  currentRoute: string;
  currentStop: number;
  inventory: string[];
  mekikiScore: number;
  hintsUsed: Record<string, number>;
  completedRoutes: string[];
  zukan: { items: string[]; npcs: string[] };
}
```

ルート・NPC・アイテムは**すべてJSONデータ駆動**にする。エンジンとコンテンツを完全分離し、Phase 1以降のルート量産（＝コンテンツを足すだけ）に備える。これが量産体制の生命線。

# 第3部 技術・実装計画

## 3.1 技術スタック（Phase 0）

| 層 | 選定 | 理由 |
|---|---|---|
| フロント | React + TypeScript + Vite（PWA） | Kaiの委任開発スタイルと相性最良。配布はURL1本、テストプレイ即日 |
| 状態管理 | Zustand | 軽い。ゲームステートに十分 |
| アニメ | Framer Motion + CSS | 絵巻スクロール・交換演出に十分 |
| 音声 | Web Speech API | 読み上げモード。追加コストゼロ |
| 保存 | localStorage | Phase 0はサーバー不要。**バックエンドを作らない** |
| ホスティング | GCP Cloud Run or Firebase Hosting | 既存のDeployBar/GCP運用に乗る |

- Phase 1でSupabase導入（アカウント・図鑑同期・デイリー行商）
- Phase 1後半でCapacitorでiOS/Androidにラップ（PWA資産をそのまま流用）→ ストア配信
- Phase 3（共有ワールド）は権威サーバー必須の別設計。手応えを見てから

## 3.2 リポジトリ構成

```
warashibe/

│   ├── SPEC.md            ← この文書
│   └── design-doc.md      ← 企画書v0.1
├── src/

│   │   ├── exchange.ts    ← accepts判定・hint_ladder進行
│   │   ├── recipe.ts      ← 組み合わせ判定
│   │   └── save.ts

│   │       └── dialogue.json

│   └── components/
└── tests/
    └── engine/            ← 交換ロジックの単体テスト（全ルート踏破の自動検証）
```

- `tests/engine` に「全ストップがヒント梯子込みで必ずクリア可能」を検証する自動テストを置く。コンテンツを足すたびに詰みチェックが回る

## 3.3 実装順序（Claude Codeへの指示単位）

1. **エンジン**: exchange.ts + recipe.ts + 単体テスト（UIなしでロジック完成）
2. **データ**: kibi-01のJSON一式（本仕様2.3を転記）
3. **画面骨格**: S2マップ + S3会話 + S4インベントリ（仮素材）
4. **交換フロー結合**: 提案→判定→ヒント→成立→価値メーター
5. **演出**: 成長チェーン・結果カード生成（html2canvas）
6. **仕上げ**: ふりがな・読み上げ・チュートリアル・音

各ステップが独立して動作確認できる。SPEC駆動の親子セッション運用にそのまま載る粒度。

# 第4部 事業設計

## 4.1 収益モデル

| フェーズ | モデル | 価格感 |
|---|---|---|
